Phase Objective

Phase 5 is the compliance check. You have built the portfolio — now you must prove it holds up. Run the fully constructed portfolio through the Bear Case scenario from Phase 1. Calculate three performance metrics — Portfolio Beta, Standard Deviation, and Sharpe Ratio — and confirm the Bear Case loss stays within each client's IPS drawdown tripwire. If a portfolio fails, you must re-allocate, document every change, and verify the fix before Phase 6.

1

The Three Required Performance Metrics

Calculate all three for each client — Base Case and Bear Case

These three metrics are the quantitative backbone of your Phase 6 presentation. Each answers a different question the Investment Committee will ask. You must calculate them, interpret them, and explain clearly how the Bear Case values differ from the Base Case — and why that difference matters for the client.

Portfolio Beta (βp)
Systematic / Market Risk
βp = Σ (wi × βi)

The weighted average of each security's individual beta. Measures how much the portfolio moves relative to the market — S&P 500 has a beta of exactly 1.0 by definition.

  • β < 1.0 → less volatile than the market
  • β = 1.0 → moves with the market
  • β > 1.0 → amplifies market swings
IPS check: A risk-averse client (high A) should have βp well below 1.0. If your βp contradicts the client's stated risk profile, the Investment Committee will ask why. Be prepared to defend it or re-allocate.
Std Dev (σp)
Total Portfolio Risk
σp = SQRT(MMULT(
 TRANSPOSE(w),
 MMULT(CovMatrix, w)))

Total risk — both systematic and idiosyncratic. This is the same formula used in the Phase 3 Solver. In Phase 5, recalculate it with the Bear Case covariance matrix (stressed, tighter correlations).

  • Bear Case σp will always be higher than Base Case
  • Equity-to-equity correlations converge toward 1.0 in stress
  • Diversification benefit nearly disappears at the worst moment
IPS check: σp must be ≤ the client's IPS maximum standard deviation. A minor Bear Case breach caused by correlation compression should be disclosed and explained — not silently ignored.
Sharpe Ratio
Risk-Adjusted Return
S = ( E(Rp) − Rf ) ÷ σp

Return earned per unit of total risk taken. The risk-free rate (Rf) is your Cash/T-Bill CME return from Phase 3. Higher is better — but only compare Sharpe Ratios across clients with the same risk mandate.

  • Base Case Sharpe should be positive
  • Bear Case Sharpe will typically be negative — that is expected
  • The hedge should improve the Bear Case Sharpe vs. unhedged
IC Q&A alert: If the Growth client has a higher Sharpe than the Retiree, the committee will ask whether that means it is "better managed." The answer is no — it reflects a different risk mandate, not superior management skill.
⚠ Excel Requirement — No Hard-Coded Metrics

All three metrics must be formula-driven and linked to your Phase 3 CME Matrix. If your instructor changes one CME input cell, βp, σp, and the Sharpe Ratio must update automatically. If nothing changes when an input is altered, you have hard-coded your audit — this is an automatic deduction.

The Crash Test — Running the Bear Case

Apply Phase 1 Bear Case CME inputs to the fully constructed portfolio

The crash test is not a new model — it is the Phase 3 Solver model re-run with Bear Case inputs. The difference in Phase 5 is that you are now applying that stress scenario to the actual Phase 4 portfolio, including its specific security betas and the derivative hedge, not just abstract asset class weights.

What Changes in the Bear Case Run

→
Swap These Inputs

  • CME Matrix: switch from Base to Bear Case expected returns
  • Correlation matrix: use the stressed version (equity correlations → 1.0)
  • Fed Funds Rate: use the Red Zone rate from Phase 1 Sensitivity Matrix
  • Consumer Sentiment: use the low Bear Case level from Phase 1

≡
Keep These Fixed

  • Portfolio weights — locked from Phase 4
  • Individual security betas — real market betas, not scenario-dependent
  • Hedge structure — strike, expiration, and contract count from Phase 4
  • AUM — $500,000 throughout all scenarios

The Four-Outcome Decision Tree

Crash Test Result Required Action Where It Appears in Phase 6
✓ PASS — within IPS drawdown limit Document the result. Confirm hedge was effective. Proceed to Phase 6. Presented as fiduciary compliance evidence in the IC Memo Risk section
✗ FAIL — unhedged exceeds limit Apply the Phase 4 hedge and recalculate. If the hedged portfolio passes, document the before/after. This is the expected outcome for most teams. The problem → solution narrative is a graded element of the oral presentation
✗ FAIL — hedged portfolio still exceeds limit Re-allocate: reduce equity weight, increase cash or short-duration bonds. Recalculate until passing. Document every change with rationale. Show the trade-off explicitly: "We sacrificed X% of upside to satisfy the IPS constraint"
σp breach — minor, stress-period only Disclose in writing. Explain it as a Bear Case anomaly from correlation compression, not a structural error. Include in the AI Audit Log as a known limitation reviewed with the AI Red Team

The AI Audit Log

Required appendix — documents every AI interaction and the team's response

The AI Audit Log is not optional. It is a graded deliverable that provides evidence your team used AI as a sparring partner — not a ghostwriter. For every phase where AI assisted, the log must record the prompt used, the AI's output, and how the team responded: accepted, rejected, or modified.

The Hallucination Clause

Any AI-generated number — beta, correlation, return estimate, option premium — that was not verified against a real data source (FactSet, Bloomberg, SEC filing, CBOE) is treated as professional negligence. The Audit Log is your proof that verification happened. A number in your submission without a cited source will be flagged.

Minimum Entries Required

Phase What to Log Evidence of Human Review
Phase 1 AI regime identification prompt and output Did the team's view agree or disagree? What specific data point resolved the conflict?
Phase 3 Any AI assistance with CME return estimates or correlation assumptions Were estimates verified against FactSet @ECO or FRED historical data?
Phase 4 Any AI-generated security analysis or derivative pricing Were betas, P/E ratios, and option premiums confirmed in a real data source before use?
Phase 5 Any AI used to interpret stress test results or frame the Bear Case narrative Did the AI's risk framing align with the Phase 1 macro thesis? Where did it diverge?

Phase 5 Submission Checklist

Complete all items for all three clients before proceeding to Phase 6
  • Portfolio Beta calculated for all 3 clients — weighted average of Phase 4 security betas. Base Case and Bear Case both reported.
  • Portfolio Std Dev calculated for all 3 clients — MMULT/TRANSPOSE formula with the covariance matrix. Bear Case uses stressed (tighter) correlation inputs.
  • Sharpe Ratio calculated for all 3 clients — Base Case and Bear Case. Risk-free rate is the Cash/T-Bill CME return from Phase 3.
  • Bear Case drawdown confirmed — hedged portfolio loss is within each client's IPS drawdown tripwire. All three clients tested and documented individually.
  • Any σp breach disclosed — if Bear Case σp exceeds the IPS maximum, it is written up with an explanation of the stress-period cause. Not silently omitted.
  • Failed portfolios re-allocated — if a hedged Bear Case still exceeds the IPS tripwire, the re-allocation is documented with before/after metrics and the upside trade-off explained.
  • Audit Scorecard completed for all 3 clients — one per client, all metrics against IPS limits with Pass / Warn / Disclosed status.
  • All metrics are formula-driven — changing any CME input cell automatically updates βp, σp, Sharpe, and drawdown. No hard-coded numbers in the audit model.
  • AI Audit Log complete — minimum one entry per phase where AI was used. Each entry includes the prompt, the output, the verification step, and the team's decision.
